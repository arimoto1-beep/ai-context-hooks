# 索引メモのフォーマット

---

## 索引メモとは

AIがあとから原文に戻るための入口です。

結論を書くためのものではなく、「どこに何があるか」「どう扱うべきか」を示すためのものです。

---

## 基本項目

### `id`

索引メモの識別子です。

形式の例：`ctx-YYYYMMDD-NNN`

一意であることが重要です。原文ファイルや日付と紐付けやすい形式を推奨します。

---

### `source`

どの原文に戻るかを示します。

- 会議ログのファイル名と時刻範囲
- Slackのチャンネル名と日時
- メールの件名と送信日
- note記事のURLと段落

原文に戻れない索引メモは意味を持ちません。必ず記載してください。

---

### `topic`

何の話かを示します。

短く、検索しやすい言葉で書きます。

---

### `related`

関連しそうな機能、資料、過去メモなどを示します。

索引メモ同士の関連を示すのにも使います。

---

### `labels`

この話題の性質を示します。

後述の候補リストから選んでください。複数指定できます。

---

### `confidence`

この索引メモの内容がどれくらい確からしいかを示します。

- `high`：原文に明確な記述がある
- `medium`：原文から読み取れるが、解釈が含まれる
- `low`：推測・ニュアンスの読み取り・確認が必要

迷ったときは `low` にしてください。`high` はよく確認してから使います。

---

### `hook_note`

原文へ戻るための短いメモです。

断定しすぎないことが重要です。

- 「〜という方向の会話があった」
- 「〜という懸念が出ていた」
- 「〜については今回は対象外という話だったが、確認が必要」

のように書きます。

---

### `ai_usage`

AIにどう扱わせるかを示します。

- `仕様として直接採用しない`
- `判断前に必ず原文を確認すること`
- `設計レビューの観点として扱う`
- `確認事項として残す`
- `未決事項として扱う`

誤用防止のための指示を明確に書いてください。

---

### `cautions`

誤用防止の注意を書きます。

- `決定事項として扱わない`
- `外部情報と自分の意見を混同しない`
- `この索引メモだけで仕様判断をしない`

特に曖昧な話題や未決事項については必ず記載してください。

---

## labels の候補

| ラベル | 意味 |
|--------|------|
| `決定` | 会議で明確に決まったこと |
| `未決` | まだ決まっていないこと |
| `懸念` | 課題・リスクとして挙がったこと |
| `対象外` | 今回のスコープに含めないと判断されたこと |
| `将来再検討` | 今回はやらないが将来的に扱う可能性があること |
| `要望` | 誰かが「欲しい」と言ったこと（決定ではない） |
| `設計懸念` | 実装上の懸念・注意点として出てきたこと |
| `確認待ち` | 別途確認が必要なこと |
| `却下案` | 検討されたが採用しないことになった案 |
| `雑談` | 参考情報・脱線・背景として出た話 |

---

## confidence の考え方

```text
high   → 原文に「〜にする」「〜と決めた」など明示的な決定がある
medium → 「〜の方向で」「〜が望ましい」など、決定に近いがやや曖昧
low    → 「〜できるといいね」「気になる」程度のニュアンス
```

「決定」ラベルでも `confidence: medium` にすることはあります。
たとえば「そういう方向で進める」という合意があっても、文章に明示されていなければ `medium` が適切です。

---

## ai_usage の書き方

AIへの指示として書きます。

良い例：

```text
ai_usage: 仕様として直接採用しない。この話題が関係する場合は原文 00:12:01-00:12:45 を確認すること。
```

```text
ai_usage: 設計レビューの観点として扱う。要件として確定したわけではないため、タスク化する前に確認が必要。
```

悪い例：

```text
ai_usage: 参考にする
```

→ 何をどう参考にするか不明確です。

---

## 決定事項・未決・懸念を混同しないための注意

### やってはいけないこと

1. **未決のものを断定して書く**

   ```text
   # 悪い
   hook_note: CSV出力は将来対応予定。
   ```

2. **懸念を仕様として書く**

   ```text
   # 悪い
   hook_note: 出力形式は拡張可能な設計にする。
   ```

   （「設計として気になる」という懸念を、「設計方針」として書いてしまっている）

3. **対象外を「不要」と書く**

   ```text
   # 悪い
   labels: 不要
   hook_note: CSV出力は不要と判断された。
   ```

   （「今回は対象外」と「不要」は意味が違います）

### 推奨する書き方

```text
# 良い
labels: 今回対象外 / 将来要望の可能性 / 設計懸念 / 未決
confidence: low
hook_note: CSV出力については今回は対象外という方向の会話。ただし不要と確定したわけではなく、将来の要望として出てくる可能性が話題に上がっている。出力形式の拡張設計についても少し言及があった。
ai_usage: 仕様として直接採用しない。判断が必要なときは必ず原文を確認すること。
cautions: 「今回対象外」を「永続的に不要」と解釈しないこと。
```
